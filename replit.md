# LifeOS (Tech-Tate)

A personal life operating system — tracks habits, calendar, finance, learning (Academy), Bible study, and couples data, with a rules-based automation engine that evaluates triggers and queues actions across the user's data.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 5000)
- `pnpm --filter @workspace/lifeos run dev` — run the LifeOS frontend
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- Required env: `DATABASE_URL` — Postgres connection string; `CLERK_SECRET_KEY` / `CLERK_PUBLISHABLE_KEY` / `VITE_CLERK_PUBLISHABLE_KEY` — Replit-managed Clerk auth

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- API: Express 5
- DB: PostgreSQL + Drizzle ORM
- Validation: Zod (`zod/v4`), `drizzle-zod`
- API codegen: Orval (from OpenAPI spec)
- Build: esbuild (CJS bundle)
- Auth: Replit-managed Clerk (`@clerk/react` on the `lifeos` frontend, `@clerk/express` on `api-server`)

## Where things live

- `artifacts/lifeos` — React + Vite frontend (pages, components, Clerk auth wiring in `src/App.tsx`)
- `artifacts/api-server` — Express API (`src/routes/*` — one file per domain: habits, calendar, automation, metrics, hubs, notifications, admin, logs)
- `artifacts/api-spec` — OpenAPI spec + Orval-generated hooks/Zod schemas consumed by the frontend
- DB schema: `packages/db` (Drizzle)

## Architecture decisions

- Migrated from a Lovable.dev/Supabase app to the pnpm-workspace stack: all client-side Supabase calls were replaced with Orval-generated API hooks against the Express API server.
- Auth moved from Supabase Auth to Replit-managed Clerk; `requireAuth`/`getAuth` middleware on `api-server` gates all `/api/*` routes.
- All 18 original automation/AI functions (rule evaluation, ultra-score calculation, calendar autofill, notification generation, conflict resolution, etc.) were ported as Express route handlers under `/api/automation/*`, `/api/system/*`, `/api/calendar/autofill`, etc.
- **Tech-Tate schema migration (2026-07-06):** replaced the multi-tenant projects/tasks/ultra-metrics schema with a single-user Tech-Tate schema. Dropped tables: `ultra_metrics`, `ultra_domains`, `state_warnings`, `system_state_daily`, `automation_context_cache`, `tasks`, `projects`, `memberships`, `tenants`, `auto_actions`. Removed `tenant_id`/`tenantId` from all surviving tables (this app is single-user; multi-tenancy concept is gone). Added six new schema domains: `finance` (budgets, income, expenses, savings_goals, debts, investment_entries, net_worth_snapshots), `academy` (topics, lessons, lesson_progress, quizzes, quiz_questions, quiz_attempts), `gamification` (challenges, challenge_completions, xp_events, badges, user_badges), `bible` (bible_verses — global content, no RLS), `couples` (couples, partner_links, couple_discussion_prompts), `subscriptions`. All new user-owned tables have real Postgres RLS (`current_setting('app.current_user_id', true)`) via a `withUserContext` transaction helper in `lib/db/src/index.ts`; shared catalog tables (badges, challenges, lessons, quizzes, quiz_questions, topics, bible_verses, couple_discussion_prompts) intentionally have no RLS. No CRUD routes exist yet for the new tables — `withUserContext` is wired but unused until those routes are built. `calendar_entries` gained bill columns (`amount`, `due_day`, `is_autopay`, `category`) instead of a separate `bills` table. Seeded the 4 hubs (Finance, Academy, Bible, Couples) into the `hubs` table.
- **Frontend dead-code cleanup after the Tech-Tate migration (2026-07-06):** the LifeOS web frontend and lifeos-mobile app had several places still calling endpoints/fields removed by the schema migration (ultra-score, auto-actions, state-warnings, AI weekly/monthly/daily insight generation — none of these have backing `api-server` routes). All were fixed: `Dashboard.tsx`, `Automation.tsx`, `CrossModuleAnalytics.tsx`, `AICoachCard.tsx`, and `Insights.tsx` (web) now either use the real `{rulesEvaluated, actionsQueued}` shape from `/automation/evaluate` or show an explanatory placeholder instead of calling a 404 endpoint; stale `tenantId`/`tenant_id` params were removed from `useMetrics.ts`/`useLogs.ts`/`useCalendar.ts`/`useNotifications.ts`. Orphaned files not reachable from any route (`UltraHub.tsx`, `StatesEngine.tsx`, `AutomationRules.tsx`, `AutomationDiagnostics.tsx`, `AutomationSettings.tsx` page + its `ActionQueueViewer`/`AutomationSettings` components, `StateCard.tsx`, `UltraDomainTile.tsx`) were deleted. On lifeos-mobile: `app/(tabs)/index.tsx` had the same dead ultra-score/daily-insight calls (fixed the same way); `habits.tsx` passed a `frequency` field `HabitInput` doesn't have and a string `habitId` where a number is required (fixed); `logs.tsx` used a `description` field that doesn't exist on `LogInput` (it's `notes`, plus `logDate` is required); `Card.tsx` didn't support `onLongPress`/`onPress` (needed by `automation-rules.tsx`) — added via an internal `Pressable` fallback; `useColors.ts` had a type-unsound cast reading the `dark` palette — simplified. Additionally on lifeos-mobile, four more reachable screens (linked from the "More" menu and a live "Projects" tab) called endpoints with no backing route at all (`/insights`, `/states-engine`, `/ultra-hub`, the Projects tab + `project/[id]` detail screen calling `useListProjects`/`useGetProject`/`useListTasks`/`useCreateTask`/`useUpdateTask`) — deleted these screens, removed the Projects tab from both native and classic tab layouts, and removed their entries from the "More" menu. `automation.tsx`/`automation-rules.tsx`/`automation-settings.tsx` were verified to call real, still-existing `/automation/*` routes and were left as-is (only needed the `Card` onLongPress fix). Also fixed an IDOR in `POST /couples/:id/partner-links` (didn't verify the caller owns the target couple before creating an invite link) — now checks `couple.userAId === req.userId` before inserting. Full `pnpm run typecheck` passes across all packages after these fixes. Note: the OpenAPI spec (`lib/api-spec/openapi.yaml`) still declares an `/ultra-metrics` path with no backing route in `api-server` — that's a spec/backend cleanup item, left as-is since no frontend code calls it anymore.

- **Bill tracking calendar + Progress screen (2026-07-09):** added `amount`, `dueDay`, `isAutopay`, `category` to the OpenAPI spec for `calendar_entries` (columns already existed in the DB from the Tech-Tate migration but were missing from the spec/generated client). `Calendar.tsx` was rebuilt from a week-view time-blocking calendar into a bill tracker: bills list sorted by due day, add/edit form (`ResponsiveFormModal`), a month calendar grid marking due days, and a "This week" bills summary. The same "bills due this week" card was added to `Finance.tsx` above the tabs. The `/reports` placeholder route was replaced with a real `Progress.tsx` page (read-only): budget-vs-actual bar chart (current month, from `budgets` + `expenses`), net worth trend line chart (reusing the `NetWorthTab` recharts pattern), and savings goals progress bars. `useCalendar.ts`'s Zod schema and create/update payload mapping were extended with the four bill fields. Verified end-to-end via Clerk Backend API + curl (see Gotchas) — POST/PATCH/GET/DELETE all round-trip the new fields correctly once `api-server` was restarted to pick up the rebuilt esbuild bundle.

- **XP, streaks, and badges (2026-07-09):** wired the previously-unused gamification tables into real gameplay. Extended the OpenAPI spec with `LessonProgressResult`/`QuizAttemptResult`/`ChallengeCompletionResult` (each adds optional `xpAwarded`/`newBadges` on top of the base entity) so `POST /lesson-progress`, `POST /quiz-attempts`, and `POST /challenge-completions` can report what was earned in the same response. New `artifacts/api-server/src/lib/gamification.ts` holds the shared logic: `awardXp` (writes `xp_events`), `checkAndAwardBadges` (evaluates each badge's `criteria` jsonb — `lesson_count`/`quiz_count`/`quiz_high_score_count`/`streak` — against the user's current totals and inserts into `user_badges` if newly qualified), and `ensureDailyChallengeHabit`/`maybeRecordDailyChallengeCheckin` (auto-creates a "Daily Challenge" habit and, once a user logs 3 challenge completions on the same calendar day, records a `habit_checkins` row and bumps `habits.streak` using the *exact same* `streak + 1` / reset-to-0 formula already used by `routes/habits.ts`'s manual check-in endpoint — streaks are not reimplemented, they reuse the habits system). Wired into `academy.ts` (lesson-progress awards `lesson.xpReward` once per lesson via an already-completed guard, not on every re-save; quiz-attempts awards 15 XP for ≥70% score else 5) and `gamification.ts` (challenge-completions awards `challenge.xpReward` and calls the daily-checkin helper). Seeded 5 badges directly via SQL (First Lesson, First Quiz, Quiz Master, 7-Day Streak, 30-Day Streak) since there's no admin UI for badge management yet. Frontend: `useAcademy.ts`/`useGamification.ts` mutation `onSuccess` handlers now call a shared `celebrateGamification()` toast helper (`src/lib/gamificationToast.ts`) that shows "+N XP" and "🏆 Badge unlocked" toasts, and invalidate the user-badges/habits query keys when new badges arrive. `Profile.tsx` got a new badge grid card (locked/earned states) using `useBadges`/`useUserBadges`. Verified end-to-end via Clerk Backend API + curl with throwaway test topics/lessons/quizzes/challenges: confirmed XP is awarded once (not on duplicate lesson-progress posts), 3 same-day challenge completions bump the Daily Challenge streak by exactly 1, and the 7-Day Streak badge fires correctly once the streak crosses the threshold. All test data and the throwaway Clerk user were deleted after verification.

- **Learn/Academy tab rebuild (2026-07-09):** replaced the topic-list-only Academy page with a full Learn experience. Added `code` (unique) to `topics` and `videoUrl`/`scheduledDate` to `lessons` (schema + OpenAPI + codegen), and three new read-only endpoints: `GET /lessons` (all lessons joined with topic name/code), `GET /lessons/today` (returns the lesson scheduled for today if one exists, else falls back to the user's earliest not-yet-completed lesson by sort order, else `null` if everything is complete), `GET /lessons/{id}` (detail, 404 if missing). Seeded 20 real topics (Budgeting, Stocks, Investing, Real Estate, Entrepreneurship, Leadership, Negotiation, Business Management, Psychology, Behavioral Finance, Economics, Taxes, Insurance, Retirement, AI & Productivity, Sales, Marketing, Team Building, Wealth Creation, Stewardship) and 7 real (non-lorem-ipsum) sample lessons spread across a week (one scheduled for "today" to exercise both branches of the fallback logic; one includes a YouTube embed URL). `Academy.tsx` is now the Learn tab home: a "Today's Lesson" card (or an all-caught-up state), a search box + topic filter dropdown, and a library grouped by topic with a filled/outline circle per lesson to show completion. New `AcademyLessonDetail.tsx` (route `/academy/lessons/:id`) renders the lesson: topic badge, computed reading-time estimate (word count / 200wpm), optional video iframe, content rendered as plain paragraphs split on blank lines (no markdown library added — content is written as plain prose, matching the spec's explicit permission to skip markdown parsing), and a Mark Complete button reusing the existing XP/badge-aware `useMarkLessonProgress` mutation. Verified end-to-end via Clerk Backend API + curl: `/lessons`, `/lessons/today` (both the scheduled-lesson and fallback-to-next-unread branches), `/lessons/{id}` (200 and 404), and lesson-progress persistence (XP + badge award, confirmed idempotent-per-lesson behavior already in `gamification.ts` is unaffected). All test data and the throwaway Clerk user were deleted after verification. Full `pnpm run typecheck` passes.

- **Daily quiz engine (2026-07-09):** built a full daily-quiz flow tied to each day's lesson. Schema: added `quizType` (default `'daily'`) to `quizzes` and nullable `explanation` to `quiz_questions`. Backend: lessons now expose `quizId` (left join to `quizzes`), new `GET /quizzes/{id}` returns full quiz + questions, `POST /quiz-attempts` enforces one-attempt-per-day **at the application level only** (checks existing attempts for that quiz within the current UTC calendar day before insert, returns 409 — there is no DB unique constraint backing this), `GET /quiz-attempts` joins in `quizTitle` for history. Seeded 7 real daily quizzes (3 questions + explanations each) matched to the 7 sample lessons. Frontend: new `Quiz.tsx` (home: today's-quiz CTA, or review-if-already-attempted, or history list), `QuizTake.tsx` (one-question-at-a-time, no back navigation, progress indicator, Next/Submit, inline per-question correct/incorrect + explanation results screen, 44px-min touch targets per mobile-web requirement — this sizing only applies to the `lifeos` web app, not the Expo app), linked from `AcademyLessonDetail.tsx` ("Take the Quiz") and the existing sidebar "Quiz" nav item. XP/badge awarding on quiz attempts reuses the existing `gamification.ts` helpers. Verified end-to-end via Clerk Backend API + curl: lessons/today→quizId, quiz detail fetch, attempt submission (XP + badge awarded), same-day re-attempt correctly 409s, history returns joined quiz title. All test data and the throwaway Clerk user were deleted after verification. Full `pnpm run typecheck` passes.

- **Tech-Tate schema pivot v2 (2026-07-09):** dropped the ledger-style finance tables (`budgets`, `income`, `expenses`, `investment_entries`, `net_worth_snapshots`) added in the original Tech-Tate migration — replaced with a savings-goal-centric model. Added `savings_challenges` (a challenge tied to a `savings_goals` row, with a check-in cadence) and `shareable_cards` (auto-generated canvas-drawn milestone images, e.g. on goal completion). `Finance.tsx` was rebuilt around three tabs (Savings Goals, Challenges, Debts) plus a "bills due this week" card; `Progress.tsx` now shows savings goals + savings challenges progress only (no budget/net-worth charts). New `SavingsChallengesTab.tsx` (create/check-in UI, auto-creates a linked habit and increments its streak on check-in, same as the gamification daily-challenge pattern) and `ShareableCards.tsx` (canvas-drawn card image + Web Share API with download fallback). Old Budgets/Income/Expenses/EmergencyFund/Investments/NetWorth tab components were deleted.
- **Home tab daily Lesson→Quiz→Challenge sequence (2026-07-09):** added `lessonId` (nullable, FK to `lessons`) to `challenges` and `responseText` (nullable) to `challenge_completions`, so a daily-action `challenge` (the gamification kind, not `savings_challenges`) can be tied 1:1 to a lesson and a challenge-completion can carry a free-text reflection. `LessonWithTopic`/`GetTodayLessonResponse` now also expose `challengeId` (left join `challenges` on `lessonId`) alongside the existing `quizId`. New `DailySequenceCard.tsx` on `Dashboard.tsx` shows a 3-step locked tracker for today's lesson: Lesson (always open) → Quiz (unlocks once the lesson's `lesson_progress.completed` is true, skipped if the lesson has no quiz) → Challenge (unlocks once today's quiz attempt exists, skipped if the lesson has no linked challenge; inline textarea for the optional response, submitted via the existing `useCompleteChallenge` hook extended to accept `responseText` + an `onSuccess` callback). Seeded 7 challenges, one per existing sample lesson. Verified end-to-end via Clerk Backend API + curl: `lessons/today` returns the correct `challengeId`, and the full lesson-complete → quiz-attempt → challenge-completion (with `responseText`) chain awards XP/badges at each step and persists the response text.

## Product

- Dashboard (with a daily Lesson→Quiz→Challenge locked sequence card), Habits, Calendar (with AI autofill, now including bill tracking), Logs, cross-domain hub metrics (Finance/Academy/Bible/Couples), Progress (savings goals + savings challenges progress bars), a full Learn/Academy tab (Today's Lesson with fallback, searchable/filterable topic library, lesson detail with video/reading time/Mark Complete), a daily quiz tied to each lesson, and an Automation engine (rules, trigger events, queue, execution history, conflict resolution) with supporting Settings/Notifications/Admin/Security pages. Finance is now savings-goal-centric (Savings Goals / Challenges / Debts tabs, bill tracking, shareable milestone cards) — the original budget/income/expense/net-worth ledger tables and UI were removed. Academy lesson/quiz completion and daily challenge completions award XP and badges (streaks reuse the habits system via an auto-created habit); badges are shown on Profile. Bible/Couples/Subscriptions data models exist in the DB with RLS but have no CRUD API routes or frontend UI yet. Deferred specs (not built): market-leaderboards, curriculum-bible, ai-coach, couples-consent.
